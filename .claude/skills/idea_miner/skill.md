# Idea Miner — 아이디어 구체화 프리프로세서

`fullstack-webapp` 파이프라인의 **입력 해상도 향상 장치**다.  
한 문장짜리 아이디어도 설계 가능한 수준의 요구사항으로 끌어올리는 것이 목적이다.

## 실행 조건

다음 중 하나라도 해당하면 이 스킬을 실행한다:
- 프로젝트 루트에 `idea.md`가 존재함
- 사용자 입력이 한 문장 수준이거나 핵심 의사결정 포인트가 비어 있음
- 사용자가 `/idea_miner`를 명시적으로 호출함

## 실행 순서

### Step 1: 입력 수집

1. `idea.md`가 있으면 읽는다
2. `idea_inquiry.md`가 있으면 읽어서 **이전 답변 문맥을 유지**한다
3. 사용자가 직접 입력한 텍스트가 있으면 함께 통합한다

### Step 2: 아이디어 분석

수집한 내용을 아래 5개 관점으로 분석한다:

| 관점 | 분석 항목 |
|------|----------|
| **제품** | 해결하려는 문제, 핵심 가치, 차별점 |
| **사용자** | 타겟 사용자, 사용 맥락, 사용자 여정 |
| **기술** | 데이터 구조, 권한 모델, 외부 연동, 비기능 요구사항 |
| **비즈니스** | 수익 구조, 운영자 기능, 성장 단계 |
| **리스크** | 실패 시나리오, 예외 상황, 규정/보안 이슈 |

### Step 3: 질문 설계

분석 결과에서 **비어 있는 핵심 의사결정 포인트**를 찾아 질문을 설계한다.

질문 설계 원칙:
- 추상적 브레인스토밍이 아니라 **아키텍처/DB/API 설계로 직결되는 질문**
- 이전 답변이 있으면 그 내용을 문맥으로 활용하고 중복 질문 금지
- 한 번에 3~5개 이내로 집중해서 묻는다 (전체를 한꺼번에 쏟아내지 않는다)
- 우선순위 높은 것부터 (P0 → P1 → P2)

### Step 4: idea_inquiry.md 생성/갱신

아래 양식으로 `idea_inquiry.md`를 프로젝트 루트에 생성하거나 갱신한다.

### Step 5: 충분성 판단

아래 조건을 **모두** 충족하면 LGTM 신호를 보낸다:

| 조건 | 확인 항목 |
|------|----------|
| 문제 정의 | 해결하려는 문제가 명확한가? |
| 사용자 정의 | 타겟 사용자가 구체적인가? |
| 핵심 기능 | P0 기능이 3개 이상 확정되었는가? |
| 데이터 모델 | 핵심 엔티티(명사)가 파악되었는가? |
| 권한 구조 | 역할(사용자/관리자 등)이 정의되었는가? |

→ 충족 시: `idea_inquiry.md` 상태를 `✅ LGTM`으로 업데이트하고 다음 단계를 안내한다

```
✅ LGTM! 아이디어가 충분히 구체화되었습니다.

다음 단계를 선택하세요:
1. `/spec_check` — 사전 기획 문서(PRD, 아키텍처 초안, 와이어프레임 등)를 먼저 확인
2. `/fullstack-webapp` — 바로 구현 파이프라인 시작
```

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| `idea.md` 없음 + 사용자 입력도 없음 | `idea.md` 작성 가이드 제시 후 대기 |
| 아이디어가 매우 추상적 | 가장 단순한 해석 하나를 제안하고 확인 요청 |
| 이미 LGTM 상태인 `idea_inquiry.md` 존재 | 갱신 필요 여부 사용자에게 확인 |

## 연동

- **입력**: `idea.md`, `idea_inquiry.md` (기존 답변 유지), 사용자 직접 입력
- **출력**: `idea_inquiry.md`
- **다음 단계**: `spec_check` 또는 `fullstack-webapp` Phase 1
