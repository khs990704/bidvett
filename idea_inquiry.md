# Idea Inquiry — BidVett (Upwork 공고 이중 리스크 스크리닝 SaaS)

> 최종 업데이트: 2026-05-27
> 상태: ✅ LGTM — 다음 단계 진입 가능 (Q1~Q6 모두 확정)

---

## 아이디어 요약

Upwork 프리랜서가 Connects(지원 토큰)를 낭비하지 않도록, 공고 텍스트를 붙여넣으면 **정량 룰 엔진(백엔드)** 과 **정성 LLM 추론(OpenAI)** 이중으로 사기/먹튀 위험을 스크리닝하고 매칭 점수(1~100)와 액션 플랜을 3초 내에 반환하는 SaaS.

## 해결하려는 문제

- 2026년 급등한 Upwork Connects 비용 + AI 스팸 지원서 범람으로 프리랜서의 지원 비용 대비 수주율 급락
- 면접·고용 가능성 낮은 공고(유령 공고, 결제 미인증, 사기성 공고)에 Connects를 소모해 자본 압박
- 본인의 프로필·예산·기술 스택과 공고의 적합도를 객관적으로 판단할 도구 부재

## 타겟 사용자

- **핵심**: Upwork 글로벌 1~5년 차 주니어/미드 레벨 프리랜서
- **확장**: 공고 분석 시간이 부족한 Top Rated 고단가 프리랜서
- **사용 맥락**: Upwork 탐색 중 공고 발견 → 페이지 복사 → BidVett에 붙여넣기 → 3초 내 리포트 확인 → 지원/포기 결정

## Claude의 질문

> 아래 4개는 **DB 스키마와 API 명세 직결** 질문입니다. 답해주시면 바로 설계 단계로 넘어갑니다.

**Q1. (정량 룰 엔진 입력 수집 방식) — P0, 가장 중요**
백엔드 정량 위험도 산출에 필요한 4개 수치(클라이언트 과거 고용률, 결제 인증 상태, 과거 지출 총액, 과거 프리랜서 평점)를 어떻게 얻으시려고 하나요? 다음 중 어디에 가까운가요?

  - **(A)** 사용자가 공고 페이지 전체를 복사해서 붙여넣으면, **백엔드에서 정규식/패턴 매칭으로 자동 추출** (Upwork 공고 페이지에 이 정보가 노출되어 있다는 가정)
  - **(B)** **LLM이 본문 분석과 동시에 이 수치도 함께 파싱**해서 JSON으로 반환 (정성 + 정량을 한 번의 API 호출로 통합)
  - **(C)** 공고 본문 입력란과 **별도로 4개 필드를 사용자가 직접 입력**하는 폼 (드롭다운/숫자 입력)
  - **(D)** 기타 — 구체적으로

→ 하이브리드 최적화 방식 (프론트엔드 전처리 + OpenAI Structured Outputs)

수집 및 파싱 메커니즘:

인풋(Input) 단계: 사용자는 별도의 입력 폼을 채울 필요 없이, 업워크 공고 페이지 전체를 복사(Ctrl+A -> Ctrl+C)하여 단일 입력란에 붙여넣습니다. (Friction-Free UX 유지)

전처리(Pre-processing) 단계: 백엔드/프론트엔드 코드(JavaScript)가 입력된 전체 텍스트 중 상·하단 메뉴나 광고 등 불필요한 데이터를 잘라내고, 클라이언트 정보와 공고 본문이 포함된 핵심 구역만 추출하여 AI API로 전송합니다. (LLM의 토큰 과부하 및 비용 차단)

AI 파싱 단계: OpenAI의 Structured Outputs(구조화된 출력) 기능을 활용하여, AI에게 정해진 JSON 스키마(양식지)를 강제 주입합니다. AI는 뇌를 쓰지 않고 정돈된 텍스트에서 4대 수치(hire_rate, payment_verified, total_spend, rating)를 찾아내어 지정된 데이터 필드에 정확히 채워 반환합니다. (할루시네이션 및 누락 방지)

최종 룰 엔진 구동: AI가 채워준 객관적 숫자 데이터를 바탕으로, 위험 등급(CRITICAL_RISK 등)에 대한 최종 판정과 UI 제어는 백엔드 시스템이 100% Rule-based로 안전하게 처리합니다.

**Q2. (결제 → 분석 권한 변환 모델) — P0**
세 가지 결제 옵션을 분석 권한으로 어떻게 환산하시나요? 예시 형태로 답변 부탁드립니다:

  - 평생 무료 3회 = ?개의 1회용 크레딧 (계정 단위? IP 단위?)
  - $0.99 단건 = 1회용 크레딧 1개 추가?
  - $4.99 주간 패스 = 결제 시점부터 7일 무제한? 분석 횟수 상한 있음?
  - $19 월 구독 = 완전 무제한? 또는 월 N회 한도?
  - 크레딧 만료 정책이 있나요? 환불 정책은?

→ 평생 무료 3회 = 3개의 1회용 크레딧 (계정 단위 방식 채택)

이유: 앞서 PRD에서 Supabase Auth를 통한 '구글 소셜 로그인'을 단일창으로 구축하기로 확정했으므로, 구글 계정 ID(UID)를 기준으로 credit_ledger 테이블에서 3개를 지급하고 차감하는 것이 개발하기 가장 쉽고 정확합니다.

※ IP 단위는 VPN이나 공공 와이파이 환경에서 무고한 유저가 차단되거나, 악의적 유저가 VPN으로 무한 가입하는 우회 리스크가 있어 제외합니다.

$0.99 단건 = 1회용 크레딧 1개 추가 충전

결제 즉시 credit_ledger에 유료 크레딧 1개가 즉시 가산됩니다. 유저가 원할 때 언제든 소모할 수 있는 단건 스팟성 옵션입니다.

$4.99 주간 패스 = 결제 시점부터 7일(168시간) 무제한 (단, 주간 100회 소프트 캡 제한)

이유: 완전 무제한으로 풀 경우, 한 계정을 여러 명이 돌려쓰거나 크롬 익스텐션 등 매크로 프로그램을 연동해 업워크의 수천 개 공고를 통째로 긁어버리는 '헤비 스크래핑 어뷰저'가 발생하여 창업자님의 AI API 비용이 파산할 수 있습니다.

일반적인 프리랜서가 일주일에 공고를 100개 이상 딥분석하는 경우는 사실상 불가능하므로, "7일간 무제한이지만 API 비용 방어를 위해 최대 100회까지 지원"하는 소프트 캡(안전장치)을 백엔드에 걸어둡니다.

$19 월 구독 = 30일 무제한 정기 구독 (단, 월간 500회 소프트 캡 제한)

이유: 에이전시(팀 단위) 유저들이 계정 하나를 공유해 한 달에 수천 번씩 돌리는 리스크를 차단합니다. "한 달간 무제한 이용 가능, 최대 500회 제한"으로 명시합니다. 500회는 매일 16~17개의 공고를 쉬지 않고 한 달 내내 정밀 분석해야 채울 수 있는 숫자로, 정상적인 1인 프리랜서에게는 완벽한 무제한과 다름없는 스펙입니다.

크레딧 만료 정책: 구매 형태에 따른 이원화 운영

단건 결제($0.99): 만료 기한이 없습니다(평생 소장). 소액 유저의 심리적 장벽을 낮추기 위함입니다.

패스 및 구독($4.99, $19): 해당 기간(7일 / 30일)이 종료되는 분(Minute) 단위 시점에 시스템이 자동으로 권한을 회수하거나 갱신(구독)합니다.

환불 정책: 글로벌 SaaS 표준(Stripe 기반 디지털 재화 규정) 적용

원칙: 구매 후 단 1회라도 분석권을 사용했다면 환불 불가 (AI API 비용이 이미 소비되었기 때문).

예외: 결제 후 7일 이내에 분석 기능을 단 1회도 사용하지 않은(0회 소모) 순수 미사용 결제 건에 한해서만 Stripe 대시보드에서 100% 전액 환불 가능. 이 내용을 약관에 한 줄 명시하여 CS 리스크를 원천 차단합니다.

> 📌 **PIVOT-01 (2026-05-29)**: 결제 프로바이더를 **Stripe → Dodo Payments**로 전환. 환불 정책(0회 사용 + 7일 이내만 100% 환불, 1회 사용 시 환불 불가)과 가격/소프트 캡 동일하나 운영은 Stripe Dashboard 대신 **Dodo Dashboard**에서 수동 처리한다. Dodo는 Merchant of Record이므로 VAT/GST 자동 처리. 상세는 `_workspace/00_input.md §11`(PIVOT-01) 및 `spec/` rev 2 참조.

**Q3. (사용자 프로필 데이터 구조) — P0**
매칭 스코어링의 3개 매트릭스(기술 일치도 40% / 예산 일치도 30% / 맥락 매칭 30%)를 산출하려면 프로필이 어느 정도 구조화돼야 하는데, 어떻게 입력받으시겠어요?

  - **(A)** 자유 텍스트 한 덩어리 (이력서 요약 텍스트만 받고 매칭도 LLM이 알아서)
  - **(B)** **구조화 폼** — 핵심 스킬셋(태그), 경력 연차, 희망 시간당 단가($), 가능 시간대, 선호 국가 등 별도 필드 + 자유 텍스트 보조
  - **(C)** 하이브리드 — 자유 텍스트로 받되 최초 등록 시 LLM이 구조화 필드를 추출해 사용자가 확인/수정

→ (C)안 하이브리드 방식 (자유 텍스트 입력 ➔ LLM 자동 구조화 ➔ 유저 검토 후 DB 저장)

데이터 수집 및 매칭 파이프라인 구조:

인풋(Input): 가입 최초 단계에서 유저는 기존 이력서 또는 타 플랫폼 프로필 텍스트를 단일 Textarea에 통째로 복사·붙여넣기 한다.

AI 구조화 필터링 (Parsing): 백엔드는 해당 텍스트를 gpt-4o-mini로 전송, Structured Outputs를 통해 아래의 스키마로 데이터를 즉시 추출한다.

skills: 배열 (예: ["React", "Node.js", "TypeScript"])

years_of_experience: 정수 (예: 3)

target_hourly_rate: 정수 (예: 40)

timezone: 문자열 (예: UTC+9)

UI 렌더링 및 유저 확정 (Validation UI): AI가 반환한 JSON 데이터를 프론트엔드 shadcn/ui 컴포넌트(Tag Input, Number Input 등)에 자동 매핑하여 유저에게 시각적으로 보여준다. 유저가 오차를 최종 수정한 뒤 [저장]을 누르면 Supabase DB의 users_profile 테이블에 정규화된 데이터로 영구 저장된다.

매칭 점수 스코어링 활용: 매 공고 분석 시, 원본 이력서 대신 DB에 저장된 이 4대 정형 데이터 구조만 공고 텍스트와 비교 연산하므로 토큰 비용을 최소화하고 매칭 정확도(기술 40% / 예산 30% / 맥락 30%)를 확정적으로 산출한다.

**Q4. (관리자/운영 기능 범위) — P1**
MVP 단계에서 운영자(=대표님 본인)가 필요한 기능은?

  - **(A)** Supabase Dashboard만으로 충분 (별도 어드민 UI 없음, 직접 쿼리로 운영)
  - **(B)** 최소한의 운영 페이지 필요 — 어떤 항목? (사용량 통계 / 사기 신고 공고 검토 / 환불 처리 / 프롬프트 버전 관리 / 기타)
  - **(C)** 기타

→ (A) 기성 대시보드 조합형 무개발 운영 방식 (No-Code Admin Stack)

핵심 기조: 1인 창업(바이브 코딩)의 출시 속도를 극대화하기 위해 MVP 단계에서는 별도의 관리자 페이지(Admin UI)를 일절 개발하지 않는다. 대신 Supabase와 Stripe가 제공하는 세계 최고 수준의 기성 대시보드를 활용하여 운영 공수를 0으로 줄인다.

운영 항목별 세부 메커니즘:

유저 관리 및 사용량 통계 (Supabase Data Browser)

개발 팀은 users_profile 및 credit_ledger 테이블을 생성하는 것으로 역할을 제한한다.

운영자는 Supabase 내장 데이터 브라우저를 통해 실시간으로 유저의 가입 현황과 크레딧 잔여량을 엑셀 형태로 모니터링하며, 필요시 GUI 상에서 데이터를 즉시 수정하거나 CSV 파일로 내보내어 통계를 확인한다.

결제 확인 및 즉시 환불 처리 (Stripe Dashboard 백오피스)

어드민 UI 내 환불 버튼 구현을 전면 취소한다.

환불 요청 발생 시 운영자가 Stripe 공식 대시보드에 로그인하여 해당 유저의 이메일 검색 후 [Refund] 버튼을 누르는 방식으로 수동 처리한다.

단, Stripe Webhook이 정상 연동되어 Stripe에서 환불이 완료되면 Supabase DB의 크레딧 잔여량도 자동으로 회수되도록 백엔드 동기화 로직은 반드시 구현한다.

프롬프트 버전 관리 (Next.js 환경변수 또는 Supabase 단일 테이블)

프롬프트 튜닝을 위한 관리자 웹 인터페이스는 개발하지 않는다.

Next.js의 환경변수(.env) 파일에 프롬프트 전체를 상수로 관리하여 Vercel 배포 시 업데이트하거나, 실시간 변경이 필요할 경우 Supabase 내에 system_prompts 단일 테이블을 개설하여 운영자가 데이터 브라우저에서 직접 텍스트를 수정(Update)하도록 처리한다.

사기(Scam) 신고 공고 검토 및 CS 피드백

analysis_logs 테이블에 is_reported(Boolean) 및 report_reason(Text) 필드를 추가한다.

유저가 화면에서 [신고하기]를 누르면 해당 필드가 업데이트되도록 구현하며, 운영자는 Supabase 대시보드에서 is_reported = true인 행만 필터링하여 일일 단위로 모니터링하고 추후 프롬프트 고도화에 반영한다.

어드민 UI 고도화 트리거 조건 (Next Phase):

일일 활성 유저(DAU)가 500명을 초과하여 수동 데이터 필터링이 불가능해지는 시점, 또는 CS 및 운영 업무를 외부 인력(알바 및 직원)에게 위임하여 Supabase/Stripe 직접 접근 권한을 제한해야 하는 시점에 정식 관리자 페이지(B안) 개발에 착수한다.

**Q6. OpenAI API 장애 시 정책**
분석 도중 OpenAI API가 실패하면 — (A) Connect/크레딧 차감 안 함, (B) 자동 재시도 후 실패하면 무료 재분석권 1회 발급, (C) 기타

→ 하이브리드 예외 처리 (성공 시 차감 + 백엔드 가시적/비가시적 자동 재시도)

예외 처리 파이프라인 구조:

크레딧 선검증 (Pre-check): 유저가 분석 요청 시, 백엔드는 크레딧 잔여량만 체크하고 차감 프로세스는 대기(Hold)시킨다.

서버단 자동 재시도 (Silent Retry): OpenAI API 통신 중 에러(5xx 계열 및 Timeout) 발생 시, Next.js 서버리스 함수 내부에서 최대 3회까지 백엔드 자체적으로 즉시 자동 재시도를 수행한다. 이 과정은 유저 화면에 노출되지 않으며 로딩 스피너만 유지된다.

최종 실패 시 처리 (Fail-safe): 3회 재시도 모두 실패 시, API 호출을 중단하고 유저 화면에 "OpenAI 서버 일시 오류" 안내 문구를 노출한다. 이때 credit_ledger 테이블의 크레딧은 단 1개도 차감하지 않는다 (0개 차감).

성공 시 최종 차감 (Deduct-on-Success): OpenAI API가 정상적인 JSON 데이터를 반환하고, 시스템이 이를 파싱하여 유저 화면에 리포트를 렌더링하는 시점에만 크레딧 1개를 최종 차감(Update) 처리한다.
