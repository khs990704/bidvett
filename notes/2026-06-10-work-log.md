# 2026-06-10 작업 로그 — bidvett

## 1. Google 로그인 후 `DEPLOYMENT_NOT_FOUND` 404 에러 해결

### 증상
- Google OAuth 직후 첫 화면이 Vercel의 `DEPLOYMENT_NOT_FOUND` 404
- 에러 ID: `icn1::gd4gh-...` (Incheon 엣지 리전)
- 그 상태에서 site URL을 다시 입력하면 로그인 정상 동작

### 원인
- Vercel 프로덕션 환경변수 `NEXT_PUBLIC_APP_URL`이 죽은 URL을 가리키고 있었음
- OAuth 콜백 라우트가 그 값으로 `NextResponse.redirect()` → 404
- 하지만 같은 응답에 `Set-Cookie`도 함께 박혀서, 직접 도메인 입력 시엔 세션 쿠키가 살아있어 로그인 상태로 보임

### 조치
- `src/app/api/auth/callback/route.ts`에서 `publicEnv.NEXT_PUBLIC_APP_URL` → `url.origin` (요청의 origin)으로 변경
- 클라이언트(`window.location.origin`)와 서버가 항상 같은 도메인을 가리키도록 통일
- 로컬/프리뷰/프로덕션 어디서든 자동 동작, env 의존 제거
- `GoogleSignInButton.tsx`는 이미 `window.location.origin` 사용 (이전 커밋 `7e68562`)

## 3. History 페이지 "Something went wrong" 토스트 원인 추적

### 증상
- 분석 기록이 없는 신규 사용자가 `/dashboard/history` 진입 시
- 빈 상태 카드 + `"Could not load history. Something went wrong. We've been notified."` 토스트 동시 표시

### 디버깅 경로
1. 처음엔 클라이언트 UI 분기 문제로 가정 → `HistoryList.tsx`에 에러 카드 추가 시도 → **잘못된 방향 (사용자 피드백)**
2. UI는 원복하고 API 응답 확인
3. DevTools Network: `/api/analyses` → 500
   ```json
   { "error": { "code": "ERR_INTERNAL",
                "details": { "reason": "db_select_failed" } } }
   ```
4. Vercel 로그에서 실제 Postgres 에러 확인:
   ```
   [analyses.list] {
     code: 'PGRST205',
     message: "Could not find the table 'public.analyses' in the schema cache"
   }
   ```

### 진짜 원인
**프로덕션 Supabase에 마이그레이션이 한 번도 적용되지 않음.** `public.analyses` 테이블 자체가 존재하지 않음. 따라서 history뿐 아니라 다른 API(analyze, credits, profile 등)도 모두 깨져 있는 상태일 것.

### 해결 (예정 작업)
Supabase 대시보드 SQL Editor에서 다음 4개를 순서대로 실행:
1. `supabase/migrations/0001_init_schema.sql`
2. `supabase/migrations/0002_rls_policies.sql`
3. `supabase/migrations/0003_triggers_and_rpc.sql`
4. `supabase/migrations/0004_seed_system_prompts.sql`

검증 쿼리:
```sql
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'public' ORDER BY table_name;
```
`analyses`, `users_profile`, `credit_ledger`, `dodo_events`, `system_prompts` 등이 모두 보여야 정상.
