## 1. `Invalid server environment: DODO_API_KEY/DODO_WEBHOOK_SECRET` 디버깅

### 증상
- "Extract with AI" 실행 시 `[unhandled] Error: Invalid server environment: DODO_API_KEY: Required; DODO_WEBHOOK_SECRET: Required`
- 스택: `src_lib_openai_prompts_ts_...` (prompts 로딩 경로)

### 원인
- `.env.local`이 PIVOT-01 (2026-05-29) 이전 상태에 멈춰 있어 `STRIPE_*`만 존재, `DODO_*` 누락
- `src/lib/env.ts`의 `serverEnv()`는 일부만 쓰더라도 schema 전체를 한 번에 검증 → DODO 키 누락이 전체 검증 실패로 번짐
- `src/lib/openai/prompts.ts:86`에서 DB/KV 캐시 둘 다 실패 시 fallback으로 `serverEnv().SYSTEM_PROMPT_VERSION` 접근 → 트리거

### 조치
- `.env.local`에 `DODO_API_KEY`, `DODO_WEBHOOK_SECRET` 추가
- 잔존 Stripe 키 정리 권장 (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_*`)
- Vercel에는 이미 등록되어 있으므로 `vercel env pull .env.local`로 동기화도 가능

## 2. Dodo Payments Webhook 핸들러 완성

### 발견한 핵심 결함

공식 SDK 타입(`node_modules/dodopayments/resources/webhook-events.d.ts`) 확인 결과:

1. **이벤트 ID 위치 오류**: Dodo `WebhookPayload`에는 `id` 필드가 없음. 멱등 키는 Standard Webhooks `webhook-id` **헤더**. 기존 코드는 verified body에서 `evt.id`를 요구 → 실제 환경에서 모든 webhook이 `Verified payload is not a valid Dodo event envelope` 로 실패하던 상태.
2. **데이터 래퍼 오해**: 실제 페이로드는 `data` 자체가 Payment/Subscription 객체. `data.object` / `data.payment` 같은 래퍼 없음. `customer_id`도 `data.customer.customer_id` 경로.
3. **유니크 제약 충돌 가능성**: `payment.succeeded`와 `subscription.active`가 둘 다 subscription INSERT를 시도하면 `uniq_subscriptions_active_per_user` / `uniq_subscriptions_dodo_sub` 위반.

### 이벤트 분리 정책 (사용자 결정)

- `payment.succeeded` → **credit_single 한 건만 처리** (`credit_ledger` +1)
- `subscription.active` → weekly_pass / monthly_sub **upsert** (idempotent)
- `subscription.cancelled` → **`cancelled_at`만 기록, status는 그대로** (현재 paid 기간 보장)

### 추가 마이그레이션
- `supabase/migrations/0005_subscriptions_cancelled_at.sql` 신규 — `subscriptions.cancelled_at timestamptz` 컬럼 추가

### 변경 파일
| 파일 | 변경 |
|------|------|
| `src/lib/dodo/webhook.ts` | 실제 Dodo `WebhookPayload` 형상에 맞게 재작성. `data` 직접 사용, 이벤트 ID는 `webhook-id` 헤더에서 수령. |
| `src/app/api/webhooks/dodo/route.ts` | `event.id`를 `webhook-id` 헤더에서 추출하여 멱등성 키로 사용. |
| `src/app/api/credits/route.ts` | `cancel_at_period_end`를 하드코딩 false 대신 `cancelled_at != null`로 도출. |
| `tests/integration/dodo-webhook.test.ts` | 새 페이로드 형상과 3-이벤트 분리 정책에 맞게 재작성. |

### #7 — Subscription 해지 API + UI
**신규**: `src/app/api/subscription/cancel/route.ts`
- POST. requireUser + rate limit (IP 120/min, user 10/min)

## 5. 남은 위험 요소 (production 전 결정/조치 필요)

### 🔴 우선순위 높음

**1. 플랜 업그레이드 시 이중 청구 위험**
`subscription.active` 핸들러는 DB의 기존 active row를 `canceled`로 정리하지만 **Dodo 측 구독은 그대로 살아있음**. weekly→monthly 갈아타면 둘 다 청구됨. 대응 선택:
- (a) `/api/checkout`에서 차단 — 이미 active sub 있으면 400 `BAD_REQUEST` (`reason: 'already_subscribed'`). **권장**.
- (b) 웹훅에서 자동 해지 — reconcile 단계에서 기존 row의 `dodo_subscription_id`로 `subscriptions.update(oldSubId, { cancel_at_next_billing_date: true })` 호출. 자동이지만 Dodo API 의존도 증가.

**2. weekly_pass 해지 UI 부재**
현재 "Cancel subscription" 버튼은 `active_subscription`(monthly)에만 노출. `weekly_pass`가:
- 7일짜리 **일회성 결제**(non-recurring)면 무시 가능
- **자동 갱신 구독**이면: `CreditsResponse.active_pass`에 `cancel_at_period_end: boolean` 추가 + UI 노출 필요

> **사용자 확인 필요**: weekly_pass의 Dodo 상품 설정이 recurring인가?

### 🟡 중간

**3. 결제 실패 모드 미구독**
현재 미구독 이벤트: `subscription.failed`, `subscription.on_hold`, `subscription.expired`.
- 카드 만료/잔액 부족 시 우리 DB는 여전히 `active`로 유지됨
- 최소 `subscription.failed` 또는 `subscription.on_hold` 구독 + 핸들러 추가 권장

**4. test_mode / live_mode 시크릿 분리**
Dodo는 endpoint마다 별도 signing secret 발급. Vercel **Preview**와 **Production** 환경의 `DODO_WEBHOOK_SECRET`이 각각 test_mode/live_mode endpoint secret을 가리키는지 확인. 한쪽만 등록되면 해당 환경 webhook이 모두 sig 실패.

### 🟢 마이너

**5. `subscription.active`의 customer_id 누락 시 silent drop**
`data.customer.customer_id` 없으면 warn 로그 후 no-op. 사용자는 결제했는데 활성화 안 됨 → Sentry alert 권장 (`SENTRY_DSN` schema에 있음).

**6. Polling 종료 후 URL replace**
페이지 떠난 상태에서 `router.replace` 호출 시 콘솔 경고 가능. 무시 가능 수준.
