## 1. `DEP0169` url.parse() deprecation 경고

### 증상
```
(node:4) [DEP0169] DeprecationWarning: `url.parse()` behavior is not standardized...
```
DODO 결제 호출 직후 Vercel Functions 로그에 출력.

### 원인
사용자 코드(src/) 사용처 0건. 출처는 `node_modules/form-data@4.0.5/lib/form_data.js:8`의
```js
var parseUrl = require('url').parse;
```
의존 트리 (`pnpm why form-data`):
```
form-data@4.0.5
└─ @types/node-fetch@2.6.13
   ├─ dodopayments@1.53.2  ← 결제 호출 시 트리거
   └─ openai@4.104.0
```

### 조치
무시. CVE 발급 안 되는 단순 deprecation, 기능 영향 없음. 노이즈 제거 원하면 Vercel env에 `NODE_OPTIONS=--no-warnings=DEP0169`. `form-data`/`node-fetch` 메이저 업이 되면 자동 해소.

## 2. `ERR_PAYMENT_UPSTREAM` — test 키 + production NODE_ENV 불일치

### 증상
결제 시도마다 502 `Payment provider is temporarily unavailable.`

### 원인 (구조적 결함)
`src/lib/dodo/client.ts:20`:
```ts
environment: process.env.NODE_ENV === 'production' ? 'live_mode' : 'test_mode',
```
Vercel 배포는 항상 `NODE_ENV=production` → SDK는 `live_mode`로 초기화. 하지만 등록된 `DODO_API_KEY`는 test 모드 키 → DODO API 거절 → catch에서 502로 변환(`client.ts:71-76`).

### 조치 (사용자 결정: Option A)
DODO Dashboard에서 live mode API 키 + webhook secret 발급 → Vercel Production env로 교체 → 재배포.

### 미해결
mode를 `DODO_ENVIRONMENT` env로 명시화하는 개선은 보류. 현재 Preview/Production 환경 모두 live 모드로 동작 — 향후 Preview에서 test mode로 검증하려면 `env.ts`/`client.ts` 수정 필요. (Option B)

## 3. `DODO_PAYMENTS_API_KEY` vs `DODO_API_KEY` 혼동 정리

DODO 공식 SDK(`dodopayments@1.53.2/index.js:75`)는 `bearerToken` 미전달 시 `process.env.DODO_PAYMENTS_API_KEY`를 기본 읽음. 하지만 이 프로젝트는 `src/lib/dodo/client.ts:19`에서 `bearerToken: serverEnv().DODO_API_KEY`로 **명시 전달** → SDK 기본 경로 비활성. Vercel env 이름은 반드시 **`DODO_API_KEY`** 사용.

## 4. DODO 콘솔 webhook 테스트 — `404 No message with the ID msg_...`

### 증상
DODO Dashboard에서 webhook 테스트 시 `404: Not Found. No message with the ID msg_3FIS8vdnaBGlergp6sVdg7qptQF exists.`

### 원인
test_mode와 live_mode는 **webhook 메시지 네임스페이스가 분리**됨. 해당 `msg_id`는 test mode endpoint의 메시지였고, live 컨텍스트에서 replay 시도 → DODO 자체 API의 lookup 실패. 우리 서버까지 오지도 않은 단계의 에러.

### 조치
live mode endpoint 화면의 "Send example" 버튼으로 **새 메시지 생성**해서 테스트.

## 5. `WebhookVerificationError: No matching signature found`

### 증상

### 원인 후보

### 조치
live endpoint의 signing secret(`whsec_...`)을 다시 복사하여 Vercel Production env 갱신 + 재배포 → 통과.

## 6. `422 Product pdt_... does not exist` — product도 mode 분리

### 증상
live API 키 적용 후 결제 시도 시:
```
[dodo.checkout] Error: 422 Product pdt_0Ng0aQqWHPfVc6BEmVIAE does not exist
```

### 원인
DODO에서 **API 키 / webhook secret / product** 셋 다 test_mode/live_mode 네임스페이스가 완전 분리. `NEXT_PUBLIC_DODO_PRICE_*`는 여전히 test mode product ID를 가리킴 → live API로 lookup 실패.

### 조치
1. DODO Dashboard → Live → Products에 세 상품(credit_single / weekly_pass / monthly_sub) 신규 생성
2. Vercel Production env 갱신:
   - `NEXT_PUBLIC_DODO_PRICE_SINGLE`
   - `NEXT_PUBLIC_DODO_PRICE_WEEKLY`
   - `NEXT_PUBLIC_DODO_PRICE_MONTHLY`
3. **재배포 필수** — `NEXT_PUBLIC_*`는 빌드 시점에 클라이언트 번들에 박혀서, env만 바꾸고 redeploy 안 하면 옛 product ID로 요청 시작.

### 결과
live mode 결제 정상 동작 확인.

## 8. `42702 column reference "balance_after" is ambiguous` — RPC 버그

### 증상
analyze 첫 실행 시 Supabase RPC 에러:
```
code: '42702'
message: 'column reference "balance_after" is ambiguous'
details: 'It could refer to either a PL/pgSQL variable or a table column.'
```

### 원인
`supabase/migrations/0003_triggers_and_rpc.sql`의 `record_analysis_and_deduct` 함수가
```sql
RETURNS TABLE (analysis_id uuid, balance_after int, source text)
```
로 선언되어 있어, `balance_after`가 PL/pgSQL 함수 본문에서 **출력 변수**로 취급됨. 그런데 L111, L119에서 `credit_ledger.balance_after` 컬럼을 한정자 없이 참조 → 이름 충돌 → 모든 analyze 호출이 RPC 단계에서 실패.

### 조치
신규 마이그레이션 `supabase/migrations/0006_fix_record_analysis_and_deduct_ambiguity.sql` 작성:
- 모든 `credit_ledger` 참조에 `cl` 별칭 → `cl.balance_after`
- `subscriptions` 참조에도 `s` 별칭 (일관성)
- `CREATE OR REPLACE`로 시그니처·반환 컬럼명·로직 모두 동일하게 유지 → TS 호출부(`src/app/api/analyze/route.ts:285`, `src/lib/credits/ledger.ts:118`) 변경 불필요

### 미적용
사용자가 Supabase 대시보드 SQL Editor 또는 `supabase db push`로 적용 예정.

## 10. 메모

- Vercel은 **env 변경 후 반드시 redeploy** — 특히 `NEXT_PUBLIC_*`는 빌드 타임에 번들 박힘. 오늘 두 번 이 함정에 빠짐.
- DODO에서 모든 자원(API key / webhook secret / product / message)이 test/live 네임스페이스로 완전 분리됨. 한쪽에서 동작한다고 다른 쪽에서 동작한다 가정 금지.
- `NODE_ENV` 기반 환경 분기는 Vercel처럼 Preview/Production 모두 `production`인 환경에서는 의도대로 동작 안 함 — 명시적 env로 분리하는 게 안전.
